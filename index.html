Welcome to the Stack Overflow Careers Product UI Library. Within this document you'll find base styles, components, and views relating to the Stack Overflow Careers project.

Purpose

This library's purpose is to educate the Stack Overflow Careers team of the various methods available to them to achieve their various visual UI goals that help achieve a consistent, on-brand experience. In addition to an unified experience, this library will also help the team achieve the following subgoals:

  • Eliminate code redundancy by leveraging current styles or extending them to achieve new modules or views while staying on-brand.
  • Propogate change more readily throughout the site.
  • Create a living document that other teams, internal or external, can review to give them guidance within their projects.

Installation

The UI Kit requires that you use a single .less file to add to your .less files. Doing so will give you access to drop-in classes that will apply styles to common elements such as buttons, as well as access to a responsive grid system. You'll also get access to commonly used variables related to colors and type. Simply add this line to the top of your .less files:

{% highlight scss %} @import "lib/base.less"; {% endhighlight %}

Contributing

Scope

Before writing code to contribute to the repository, think about what is being affected with the LESS you're writing. Because we've never done this before, it may be unclear where LESS should live based on its scope. As it stands, there are three tiers of scope that determine where the LESS should live.

Scope

Where it should be

Example

Single page, single project

It's own LESS file that isn't used anywhere except that page

Hero image sizing for city pages in citypages.less

Multiple pages, single project

In the project's shared LESS file

Company logos across the site in shared.less

Multiple pages, multiple projects

Should probably live in the UI Kit, so read below

Tech tags across Careers, Q&A, etc. in the appropriate file within the UI Kit

Graduating LESS

As a general rule, you should start with implementing your LESS in the smallest scope possible - almost always at the page level - then graduate that LESS as the scope expands. This is closest to how we develop LESS anyways.

For example, I may create icons for industries on city pages, which is very specific to one feature of Careers. The relevant LESS would live in a page-specific less file like citypages.less, even if it may one day go to other parts of Careers. Perhaps after a few weeks, a new feature on Careers requires those industry icons. At this point, it's best to move the relevant code from citypages.less and into shared.less which is used across the Careers project. As a final stage, we may start thinking about using industry icons in Stack Overflow Q&A profiles, and this is when you should put the relevant code into a pull request for the UI Kit.
